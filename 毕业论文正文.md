# 基于 AI 的高校学生学习助手系统设计与实现

---

## 摘 要

高校计算机类专业普遍开设 C/C++ 程序设计课程，该语言贴近硬件与内存模型，作业与实验量大，但课下答疑资源有限：学生提交代码后往往只能得到评测机“通过/错误”的二元结果，缺少对编码规范、资源管理与算法选择的文字反馈；临近就业时，技术面试又要求临场阐述原理与手写代码，仅靠教材与视频难以形成稳定的表达节奏。与此同时，生成式大语言模型在代码理解、对话生成与结构化输出方面能力显著提升，智能体平台降低了 Bot 搭建与知识库挂载成本，为“把练习、讲评、答疑、模拟面试放在同一套系统里”提供了工程条件。

本文围绕毕业设计题目《基于 AI 的高校学生学习助手系统设计与实现》，完成可运行的 Web 原型系统 MallocMentor。系统采用 Next.js App Router 组织页面与 API Route，数据层使用 PostgreSQL 与 Prisma ORM，身份认证采用 NextAuth.js 与凭据登录；练习模块内嵌 Monaco Editor，服务端通过可配置的 Piston 或 Judge0 沙盒远程编译执行 C/C++，避免不可信代码在教学机本地进程内直接运行。AI 能力通过字节跳动扣子（Coze）发布的多个应用型 Bot 提供：面试与知识助手走 HTTP SSE 流式返回，由服务端解析扣子事件后再以 SSE 推送给浏览器；代码审查与学习路径推荐在非流式模式下拼接完整回复后解析 JSON，其中审查结果含综合分与六维能力分，并采用加权移动平均更新雷达图。当 Bot 未配置或调用失败时，面试端回退为本地 Mock 对话，路径推荐回退为基于雷达最低维度的规则模板，保证演示与调试不断链。站内另有 Markdown 知识库、五条阶梯式学习路径模板、活动日志与成就系统，用于支撑学习过程记录与轻量激励。

在自建测试环境中，对注册登录、代码运行与提交、雷达更新、流式面试、知识问答、路径推荐降级等主流程编写了可重复执行的用例并记录现象；性能上对外部沙盒与模型首包延迟进行了单独观测。结果表明，在依赖服务可用且密钥配置正确的前提下，各模块能够协同完成设计目标，为后续接入校本题库与教学管理预留了扩展点。

从方法上看，本文采用“成熟模型 + 清晰软件架构”的路线：不在论文周期内训练基座模型，而把精力放在数据模型、接口契约、容错与可观测性上，使系统具备可替换性——扣子侧可换为其他兼容 HTTP 的智能体网关，沙盒可换校内判题机，雷达权重与通过阈值可通过配置调整。从教学伦理上看，系统在文案与功能上把 AI 输出定位为辅助反馈，最终成绩认定权仍在教师与教务制度；这也是高校信息化项目常见的合规边界。

全文结构遵循学院大纲：先综述技术与需求，再给出设计与实现，最后以测试与总结收束。文中所列表名、环境变量名、API 路径与仓库源码保持一致，便于评阅人按图索骥。需要说明的是，大模型输出存在随机性，论文在测试部分已指出“宜看趋势不宜唯分数论”，部署到真实课堂时尚需结合教学大纲做二次校准。若学院对“摘要字数”单独计数，可将本摘要按 Word「工具-字数统计」核对后微调，不必与正文 Markdown 完全同长。

关键词：大语言模型；编程教育；Next.js；Coze；智能体平台；在线评测；服务器推送事件；代码沙盒；能力评估；形成性评价；检索增强生成

## ABSTRACT

University computer science programs still rely heavily on C and C++ courses that are close to hardware and memory models. Students often receive only binary judge feedback after submitting homework, while technical interviews require spoken explanations and on-the-spot coding practice that textbooks alone cannot simulate well. Recent advances in generative large language models and low-code agent platforms make it realistic to integrate automated commentary, conversational tutoring, and mock interviews into one learning web application.

This thesis implements MallocMentor, a full-stack assistant for C/C++ learners. The stack combines Next.js App Router, PostgreSQL, Prisma ORM, and NextAuth.js credentials authentication. Monaco Editor is embedded for in-browser editing; a sandbox adapter calls Piston or Judge0 over HTTP so user programs never execute inside the application process. Four Coze bots cover mock interviews, code review, knowledge Q&A, and learning-path recommendation. Streaming bots are proxied through server-side SSE; non-streaming bots return JSON-like text that is parsed after stripping Markdown fences, driving submission grading and a six-dimensional capability radar updated with an exponentially weighted moving average. Rule-based fallbacks and mock interview replies keep the UI usable when tokens are missing or upstream APIs fail. Additional features include a Markdown article store, staged learning-path templates, activity logs, and achievements.

Functional tests were executed in a local environment for authentication, sandbox runs, AI-assisted submission, streaming dialogue, and recommendation fallback. Performance observations separate database latency from third-party sandbox and model time-to-first-token delays. The prototype meets its design goals under normal external service availability and documents clear extension paths for campus-specific content. Limitations include reliance on third-party APIs and the stochastic nature of LLM grading, which should be calibrated by instructors before summative use.

Key words: large language model; programming education; Next.js; Coze; server-sent events; code sandbox; competency assessment

# 1 绪论

## 1.1 研究背景与研究意义

### 1.1.1 教学侧痛点

C/C++ 课程在计算机类专业培养方案中通常承担“建立系统观与资源观”的任务：指针、对象生命周期、STL 与并发等内容相互嵌套，学生若在基础阶段形成错误心智模型，后续操作系统、网络与嵌入式课程都会被动放大缺陷。传统教学依赖课堂讲授与实验课答疑，但班级人数偏多时，教师难以及时阅读每一份提交的中间版本；校内 OJ 或在线评测侧重判题结果，对“为何超时”“为何段错误”的解释往往要靠学生自行检索，学习效率不稳定。

### 1.1.2 就业侧需求

企业校招技术面试常见流程包含：计算机基础问答、语言特性深挖、手写代码与小项目追问。与笔试不同，面试强调边想边说，学生若缺少陪练对象，容易出现表达中断、术语不准或遗漏边界条件。线下模拟需要师兄师姐或教师投入时间，线上零散工具又难以与学习记录、能力画像打通。

### 1.1.3 技术契机与课题定位

2023 年以来，大语言模型在代码补全、解释与多轮对话上的可用度明显提高；同时扣子、文心智能体等国内平台提供可视化工作流、知识库与 HTTP 发布接口，使毕业设计在数月周期内完成“可替换模型、可演示交互”的集成成为可能。本课题不追求训练新模型，而是把成熟模型服务封装进教学软件工程链条：身份认证、数据持久化、沙盒执行、结构化评价与可视化雷达在同一应用中闭环，便于学校后续替换为私有部署模型或校本知识源。这与开题阶段对“生成式 AI 重塑教育服务形态”的判断一致；具体工程选型则以本仓库可运行实现为准，后文 **§4.1.4** 对开题报告中的技术设想与最终实现做了对照说明。

### 1.1.4 研究意义归纳

理论层面，可归纳“教育场景下多智能体分工 + 流式/非流式协同”的一种参考架构。实践层面，MallocMentor 为 C/C++ 学生提供可登录的个人中心、可追踪的练习与面试历史，以及可配置的知识扩展，减轻重复性答疑压力。工程层面，代码仓库中对接口、环境变量与降级路径有明确边界，便于二次开发与课程大作业复用。

从可复制推广角度看，本系统的价值不在于“界面多炫”，而在于把智能能力拆成可替换的适配器：沙盒适配器、模型适配器、推荐规则适配器。学校 A 可以用校内判题服务替换 Piston；学校 B 可以把扣子换成自建 vLLM，只要保持 HTTP 与 JSON 契约，业务页面改动最小。这种思路对预算有限、但希望逐步引入 AI 的院系尤其友好。

从学生发展角度看，系统强调“过程数据”而非一次性结果：活动日志、提交历史、雷达变化曲线（若后续做时间序列）都能支撑教师做学习干预。与只在考前刷题相比，持续记录更有助于形成性评价。当然，任何数据收集都需告知用途并获得授权，本文原型在正式部署前必须完成隐私评审。

从作者能力训练角度看，完成本课题等价于经历一次中小规模全栈交付：需求澄清、Schema 设计、API 设计、第三方集成、流式协议调试、测试用例编写与论文写作。这些环节在工业界同样存在，因此毕业设计不仅是“交作业”，也是向工程岗位过渡的演练。

## 1.2 国内外研究现状

### 1.2.1 国外研究现状

从人机对话技术演进看，早期规则系统（如 ELIZA）依赖模式匹配，易解释但扩展成本高；随后出现基于统计与浅层学习的对话流水线，典型架构常划分为自然语言理解、对话状态跟踪、策略学习与回复生成等模块[16]；当前以深度大模型为代表的第三代方法在表征能力上显著增强，但也更依赖数据规模与对齐策略。该脉络说明：教育场景若仅依赖“聊天窗口”，仍需要外围工程把会话与题库、画像、审计日志绑定，才能形成可管理的教学过程数据。

国外高校信息化起步早，Online Judge 与自动评分在算法竞赛训练中已成熟，研究文献常见方向包括：基于符号执行或变异的测试用例生成、基于抽象语法树的抄袭检测、以及将自然语言生成用于错误解释。近年 ACM 与 IEEE 会议中出现将 Transformer 用于编程习题提示、代码修复建议的工作，普遍强调两点约束：其一，执行用户代码必须在隔离环境（容器、虚拟机或第三方判题服务）中完成；其二，模型输出不可直接作为最终成绩，需要与测试用例或教师复核结合。

在具体平台上，DOMjudge、PC^2 等竞赛系统侧重判题与榜单；Coursera、edX 等慕课平台侧重视频与测验，编程作业常外接第三方沙盒。近年来编程学习工具链趋向“编辑器内智能”，例如将大模型嵌入 VS Code、JetBrains 系 IDE，以补全与对话形式辅助写代码。GitHub Copilot、Cursor 等产品显著提升熟练开发者的效率，但其交互范式更偏向“直接产出代码”，若缺少课堂规则与过程性评价设计，初学者容易形成依赖而弱化逐步排错能力；这与模拟面试、就业指导领域强调“反复演练与反馈”的研究结论相呼应[17][18]。这类产品对个体开发者友好，但对课程负责人而言存在数据孤岛：教师难以统一收集全班在模型辅助下的修改轨迹，也难以把“面试模拟”与“期末题库”绑定到同一套账号体系。教育研究中也讨论“AI 辅助是否削弱调试技能”的实证结果，部分对照实验表明：在明确教学设计与使用规范下，工具可缩短完成时间，但若缺少形成性评价与课堂讨论，学生可能过度依赖生成结果。

从系统架构视角，国外高校 IT 部门常采购成熟 LMS 再二次开发插件；自建全栈 Web 应用成本较高，公开论文中更常见“插件式”或“笔记本式（Jupyter）”方案。本文采用单体 Next.js 应用，把练习、面试、知识库与画像放在同一数据库中，便于小规模院系独立部署，与大型 LMS 形成互补。产品形态上，GitHub Copilot、Replit 等把模型嵌进编辑器，体验流畅，但课程若希望统一题库、统一雷达维度与教学过程数据采集，仍需自建业务系统；国外经验证明“模型有用”，却较少给出与单一课程大纲、六维能力量表紧绑定的轻量开源方案，这正是本课题可以切入的差异点。

### 1.2.2 国内研究现状

国内智慧教室、雨课堂、学堂在线、超星等平台侧重签到、课件、直播与基础学情统计；编程类则多见 OJ 与慕课组合。研究上，知识图谱推荐学习路径、学习行为分析与成绩预测等主题较多，省部级教改项目常聚焦“线上线下混合式教学”。随着文心一言、通义千问、盘古等国产大模型普及，“如何可控、安全地使用模型”成为新的关注点：提示工程（Prompt Engineering）被概括为通过结构化指令引导模型输出质量与格式的实践[19][20]，在语法严格的 C/C++ 教学中尤为关键——不完善的提示容易导致概念边界模糊、推理跳步或风格杂糅。

与此同时，大模型的“幻觉”（生成与事实不符或具误导性的内容）在编程辅助场景会被放大，例如捏造不存在的 API、错误的资源释放顺序等。学界与工业界普遍采用“外部知识 + 约束生成”的思路缓解该问题，例如在检索与生成之间引入校验与反馈循环[21]。对本课题而言，**RAG 与权威语料挂载**既可由应用侧自建向量库实现，也可由智能体平台侧知识库承担；MallocMentor 采取后者：扣子 `knowledge` Bot 挂载课程相关文档，应用侧以 HTTP/SSE 接入并保持站内结构化文章与平台知识库分工并存（详见第 2、4、5 章），从而与开题报告中“抑制幻觉、增强事实锚定”的目标一致，但落地方案与开题设想中的“自建向量数据库”并不等同。相较自建向量库方案，国产智能体平台在网络可达性、中文语料与合规上更贴近国内高校教学场景，扣子、文心智能体、通义应用等支持上传 PDF/Markdown 切片做 RAG，使应用侧以轻量集成即可完成知识增强问答演示。

国内 OJ 商业化产品与高校自研平台并存：部分学校将 OJ 与教务系统对接，实现作业截止与成绩回写；但在“自然语言讲评”“模拟面试对话”“多维度能力画像”三方面同时落地的产品仍不多。开源社区可见基于 Spring Boot 或 Django 的教学系统，但技术栈与本文 React 全栈不同，且较少见到与国产智能体平台 HTTP SSE 深度集成的完整示例。本文把多 Bot、流式对话、结构化评审与规则降级写入同一仓库，并给出环境变量模板，降低复现成本。政策与伦理上，高校通常要求课堂明确 AI 使用边界、作业声明是否允许辅助，以及保护学生数据；本系统 Token 在服务端、数据在自建库，比“学生各自安装境外插件”更易做访问审计。

当前公开仓库中，单点演示“调用 Chat API 讲题”的代码较多，而同时覆盖多 Bot 隔离、会话持久化、流式渲染、结构化成绩写库、规则降级与成就激励的完整案例仍不多见。本文在工程完整性上向前推进一步，并把设计决策写入论文，便于答辩逐项对照。

## 1.3 本文研究内容

全文工作与仓库代码一致，可概括为下列七条，每条在后文对应独立章节或小节展开。

（1）需求工程：在角色“在校学生”假设下，梳理功能需求（认证、练习、沙盒、审查、面试、知识库、路径、成就）与非功能需求（性能、安全、可扩展），并给出可行性论证。

（2）总体与数据设计：采用 B/S 三层逻辑结构，划分八大功能模块；完成概念结构说明与逻辑表设计，给出与 Prisma 模型一致的字段与索引策略。

（3）前端实现：基于 Next.js 与 React 完成路由与布局；练习页集成 Monaco；面试与知识助手消费 SSE；仪表盘用 Recharts 绘制六维雷达。

（4）后端实现：RESTful API 统一响应封装；NextAuth.js 管理会话；沙盒适配器支持 Piston/Judge0；所有写操作校验 `userId` 防止越权。

（5）AI 集成：统一接入层负责智能体请求编排与流式数据解析；业务层按场景配置专用提示策略与结构化结果映射；推荐能力在异常时自动回退规则策略。

（6）辅助机制：Markdown 文章元数据入库；学习路径模板 `LEARNING_PATH_TEMPLATES` 描述五阶段阶梯；成就模块按触发类型筛选候选键并批量查重写入。

（7）测试与评估：搭建测试环境，设计功能用例表与性能观测方法，记录外部依赖对耗时的影响，为第 6 章提供可复核描述。

## 1.4 论文结构安排

# 2 相关技术与理论基础

## 2.1 大语言模型与 Coze 智能体平台

### 2.1.1 生成式人工智能概述

生成式人工智能（AIGC）泛指能够自动产生文本、图像、音频、代码等内容的一类模型。以 Vaswani 等人提出的 Transformer 结构为基础，GPT 系列模型采用自回归方式：根据前文词元预测下一个词元的概率分布并采样输出。参数规模与训练数据量上升后，模型在代码补全、错误诊断摘要、多轮对话等任务上表现出“通识 + 模式匹配”的综合能力，但也带来三类典型风险：事实性幻觉（编造不存在的 API）、对齐问题（不当建议）与安全风险（诱导生成恶意代码）。因此在教育产品中，模型输出应被视为“建议”，与自动判题、教师评价分层管理。

从系统架构角度，调用大模型通常经历：提示构造、HTTP/gRPC 请求、流式或非流式响应解析、结果落库或展示。本课题关注的是最后一公里的软件工程：如何把响应可靠地接入已有用户体系与数据库事务，而不是讨论预训练细节。

### 2.1.2 大语言模型在编程教育中的应用

编程教育中的模型应用可粗分为三类场景。（1）解释与提示：对编译器报错做自然语言说明，或给出不泄露完整答案的思路提示。（2）静态评审：在不运行代码的情况下指出潜在空指针、资源泄漏、STL 误用等，类似加强版 linter。（3）对话式测评：模拟面试官追问 RAII、虚表布局、并发原语等。三类场景对延迟与输出形态要求不同：解释类可流式；评审类往往需要一次性 JSON 供程序解析；面试类既要流式提升体验，又要在服务端拼接全文后写入历史记录。

国际 CS1/CS2 课堂近年出现若干实证研究：部分实验表明，在教师明确规则下使用对话式工具可降低完成练习题的时间，但也观察到学生可能过度依赖生成代码而削弱调试能力。因此本系统在交互设计上把“运行沙盒”与“提交评审”分开：学生仍需亲自面对编译器与测试输出，模型主要补充自然语言层面的反馈，而不是替学生点“运行通过”。在评价伦理上，自动评分向学生公开维度（六维雷达），使其理解“为何此项偏低”，比单一分数更符合形成性评价理念。

本系统在代码提交链路中采用“结构化结果 + 容错解析”的方式：当模型输出满足格式约束时，系统将其转换为可写库的评分与建议；当输出格式不稳定时，系统保留原始文本并继续完成提交流程，避免因单点解析失败导致业务中断。该策略在教育学理上仍属于形成性评价辅助，不能替代期末统一阅卷。后续若与教务系统对接，建议将 AI 分数权重限制在平时成绩的一定比例内，并保留教师抽查与申诉渠道。

从学习科学角度，即时反馈（immediate feedback）若能解释错误原因，比单纯对错更能促进迁移；但反馈过长或过于“替学生想完”可能产生认知负荷过载或依赖。因此，系统将评审反馈组织为“问题定位 + 改进建议”的双层结构，并控制输出粒度，便于学生快速扫描重点、逐项修正，而非被冗长文本淹没。

### 2.1.3 Coze（扣子）Bot 开发平台简介

扣子（Coze）是字节跳动推出的智能体搭建平台，支持在工作流中编排插件、知识库与大模型节点，并可将 Bot 发布为独立应用，通过 HTTP 接口以 Bearer Token 访问。与本课题直接相关的特性包括：多 Bot 隔离（不同 Token 与 project_id）、会话字段 `session_id` 支持多轮上下文、以及服务端返回 `text/event-stream` 的 SSE 数据帧。

MallocMentor 将面试、代码审查、知识问答、路径推荐映射为独立智能体能力，并通过统一接入层管理请求构造、会话标识与返回解析。该设计的核心价值是“能力解耦”：当某一智能体在平台侧调整提示词或知识库时，应用层通常无需改动业务页面，仅需更新配置即可生效，从而降低维护成本。

## 2.2 Prompt 工程与结构化输出

### 2.2.1 Prompt 工程基本原理

Prompt 工程是在不改动模型权重的前提下，通过自然语言指令提高任务成功率的技术集合。常用手段包括：角色设定、输出格式约束、反例提示与少样本示例（Few-shot）。对结构化评审任务，本系统将题目上下文、测试信息与学生代码一并输入模型，并要求以固定结构返回评价结果。为兼容不同语言场景，评分维度允许出现“暂不适用”项，画像更新时对该类值执行保护性处理，避免误伤历史画像。

### 2.2.2 结构化 JSON 输出与多轮对话管理

模型在真实场景中的输出形式并不总是稳定，常出现格式包裹、字段缺失或增量片段不完整等情况。为此，系统采用“预处理 + 容错解析 + 结构校验”的三阶段策略，以提高结构化结果落库的成功率。多轮对话依赖会话标识管理，系统在首轮建立会话后持续复用同一上下文，从而保证追问连贯性。

该设计将“协议处理”与“业务语义”分离：前者由统一接入层负责流式/非流式兼容、异常隔离与重试边界，后者由各业务模块负责提示语义与字段映射，符合单一职责原则，也便于后续替换模型网关。

## 2.3 平台侧知识增强能力

### 2.3.1 扣子平台知识库功能简介

检索增强生成（RAG）指在回答问题前，先从向量库或倒排索引中检索相关文档片段，再把片段与用户问题一并送入大模型，以减少“胡编教材章节”的概率。扣子侧知识库负责切分、向量化与召回；应用侧只需保证用户问题能到达 Bot。对本课题，可将《C++ Primer》讲义节选、实验指导书、常见面试问答等导入知识库，由平台自动完成索引。

### 2.3.2 知识库在问答 Bot 中的作用

站内 `KnowledgeArticle` 表维护结构化文章（标题、slug、分类、预计阅读时长、文件路径），支撑学习路径 `articleSlug` 跳转与阅读进度统计；扣子知识库则适合承载大体积或更新频繁的原始资料。两者并存时，学生在前者按章节系统学习，在助手窗口对细枝末节提问，由 RAG 召回片段后生成回答。论文实现上，知识助手接口与站内文章接口解耦：前者走 Coze SSE，后者走 Prisma 查询，避免把大量正文重复存进关系库。

## 2.4 关键开发技术

### 2.4.1 Next.js 全栈框架

Next.js 13+ 的 App Router 将 React 服务端组件、客户端组件与 Route Handler（`route.ts`）统一在同一目录树下，支持边缘运行时与传统 Node 运行时混合部署。本课题选用 Node 运行时执行 Prisma 与长连接 SSE，避免 Edge 限制。路由以文件夹划分，`page.tsx` 负责 UI，`api/**/route.ts` 负责 JSON 或流响应，`layout.tsx` 提供共享导航。

### 2.4.2 React 与组件化 UI 开发

React 18 并发特性与 Hooks 使状态逻辑可复用。界面层采用 Tailwind CSS 原子类与 shadcn/ui 无头组件，减少手写 CSS 维护成本。练习、面试、仪表盘各自拆分为编辑器子树、消息列表子树与图表子树，父组件只负责数据获取与错误边界，降低耦合。

### 2.4.3 PostgreSQL 与 Prisma ORM

PostgreSQL 支持 ACID 事务、丰富索引与大文本字段，适合存储对话记录、题目测试信息等。Prisma 通过声明式数据模型生成类型安全客户端，迁移文件可纳入版本控制。本系统对高频过滤条件建立二级索引，避免全表扫描。

### 2.4.4 Monaco Editor 代码编辑器

Monaco Editor 提供语法高亮、括号匹配、多光标等能力，与 VS Code 一致的主题体系可降低学生切换成本。练习页将编辑器内容绑定为 React state，切换 C/C++ 语言时同步 `language` 字段给沙盒接口，保证编译器选项正确。

### 2.4.5 在线代码沙盒执行技术

沙盒执行的核心目标是隔离：用户进程不得访问宿主文件系统敏感路径、不得无限占用 CPU 与内存。Piston 与 Judge0 均通过 API 提供编译运行服务，由远端调度器限制时间与资源。本系统 `sandbox.ts` 使用 `fetchWithTimeout`（默认 15s）防止公共实例卡死拖垮 Node 事件循环；`run` 路由将编译失败与运行错误格式化为统一 `output` 字符串返回前端。

### 2.4.6 SSE 流式通信技术

SSE 基于长连接 HTTP，由服务器单向推送 `text/event-stream`，浏览器端可用 `ReadableStreamDefaultReader` 分块读取。与 WebSocket 相比，SSE 基于标准 HTTP，更易穿透部分校园网代理；缺点是仅单向。本系统面试与知识场景只需服务器推送到客户端，故 SSE 足够。实现上，服务端将扣子增量文本再包装为 `data: {...}\n\n` 帧，首帧可发送 `type: 'session'` 携带 `sessionId`，末帧发送 `[DONE]`，前端据此结束加载态。

| 技术                 | 在 MallocMentor 中的职责 |
| ------------------ | ------------------- |
| Next.js App Router | 页面路由、API Route、部署单元 |
| Prisma             | ORM、类型安全数据访问        |
| NextAuth.js        | 登录会话、密码哈希           |
| Monaco Editor      | 浏览器内代码编辑            |

从工程权衡角度补充说明：为何不选用纯 Spring Boot 或纯 Django？因指导教师与本人技术栈偏向 TypeScript，且 Next.js 能把前端组件与 API 放在同一仓库，类型定义可共享，毕设周期内迭代更快。为何不选用 WebSocket 做面试流？因当前业务只需服务器推送到浏览器，SSE 基于 HTTP/1.1 即可工作，部署与鉴权模型更简单；若未来要做双向二进制通道再升级不迟。为何 PostgreSQL 而非 MySQL？二者均可，选型依据是 Prisma 示例与托管云厂商对 JSON 与全文检索的支持习惯，本校若统一 MySQL 也可替换。为何沙盒默认 Piston？因其公共实例免 Key 便于开发；生产应评估 SLA 后改为校内 Judge0 或容器集群。为何能力雷达用六维？与 `CapabilityRadar` 模型及评审 Prompt 对齐，避免“数据库一维、提示词另一套”的割裂；维数过多学生看不懂，过少又不足以指导学习路径推荐。

教学场景下还有一个隐性需求：可审计性。PostgreSQL 中的 `ActivityLog`、`CodeSubmission`、`InterviewSession` 为教师或教务日后抽查提供依据；若仅用无状态聊天窗口则难以复盘。本文架构把“可审计”当作一等需求，而不是事后补日志。

# 3 系统需求分析

## 3.1 系统总体需求分析

### 3.1.1 目标用户与使用场景

系统默认用户为已具备基础计算机操作能力的高校本科生或同等学力学习者，主要场景包括：课前预习（阅读知识库文章）、课中实验（运行与调试代码）、课后巩固（提交练习并接受文字评审）、求职准备（模拟技术面试）、以及阶段性自评（查看雷达图与成就）。系统不假设机房预装特定 IDE，只要浏览器可用即可访问，降低机房维护成本。

### 3.1.2 业务边界说明

本版本不包含完整的教务管理（排课、成绩单的行政流程）、不包含支付与商业化运营模块，也不包含对 C/C++ 以外语言的评测。题目与文章可由种子数据或 SQL 导入维护；若学校需要教师端批量导入，可在未来增加管理员角色与 CSV 接口，当前论文不展开实现细节。

### 3.1.3 顶层用例归纳

从参与者“学生”视角，顶层用例包括：注册与登录、浏览仪表盘、管理学习路径、阅读文章并收藏、在练习页运行代码、提交代码获取评审、创建并参与模拟面试、在知识窗口提问、查看成就与活动。所有用例在登录后触发，匿名用户仅可访问登录注册页等公开路由（具体以中间件配置为准）。

## 3.2 系统功能需求分析

下列各小节采用“需求描述 + 输入输出 + 异常”三段式书写，便于在第 6 章映射测试用例。

### 3.2.1 用户管理需求

功能描述：学生使用邮箱注册账号，使用密码登录；服务端创建用户记录并对密码做不可逆哈希处理；会话由认证中间件统一维护。输入为注册/登录表单；输出为会话凭据与跳转至仪表盘。异常包括：重复邮箱注册应提示失败；密码错误拒绝登录；未认证访问受保护资源应被拦截。

### 3.2.2 编程练习与代码执行需求

功能描述：题目列表支持难度（Easy/Medium/Hard）与分类（指针、STL 等）筛选；题目详情页展示描述、标签与测试用例摘要。学生可在编辑器中修改代码后执行“运行”，由服务端调用沙盒返回编译与运行结果；执行“提交”则进入评审链路并落库。运行与提交分离，避免学生为了试探评审而频繁触发模型调用。

### 3.2.3 AI 代码审查与能力评估需求

功能描述：当模型配置可用时，系统将题目与代码组织为结构化评审输入，返回综合分、问题清单、改进建议与能力维度评分。业务规则为：综合分达到阈值记为通过，未达到则记为未通过。若结构化解析失败，系统仍保留原始评审文本并完成提交，避免流程中断。雷达更新采用加权移动平均：新值 = round(旧值 × 0.7 + 本次 × 0.3)，对“不适用”维度保持旧值。

### 3.2.4 AI 模拟面试与评估需求

功能描述：学生基于模板创建面试会话，类型分为技术类与行为类。发送消息时，系统先持久化用户输入，再调用面试智能体并以流式方式返回；对话结束后写入完整会话记录并维护统一会话标识。若智能体不可用，则回退为本地模拟问答，保证课堂演示与调试不断链。会话可标记完成，用于成就统计。

### 3.2.5 知识库学习与问答需求

功能描述：站内文章支持分类浏览、详情阅读、收藏与阅读进度；系统可记录阅读行为用于学习分析。知识助手将用户问题转发给知识智能体，并以流式方式返回答案。助手回答可引用平台侧知识片段，与站内结构化文章形成互补。

### 3.2.6 学习路径规划需求

功能描述：系统预置五条模板路径（基础、指针与内存、OOP、STL、并发），每条包含若干学习步骤。学生可拥有多条学习路径记录，并区分进行中、完成与暂停状态。推荐功能读取能力画像与历史完成情况，优先采用 AI 个性化推荐；当外部能力不可用时，回退到基于薄弱维度的规则推荐。

### 3.2.7 成就系统与用户激励需求

功能描述：成就规则由统一配置与后端检测逻辑共同维护，覆盖首次尝试、成绩里程碑、路径完成与学习连续性等场景。系统在提交、面试结束、路径推进、日常学习统计等事件上触发检测：先筛选候选规则，再批量去重，满足条件即发放成就并写入活动日志。对重复触发的成就通过唯一约束进行幂等保护。

## 3.3 系统非功能需求分析

### 3.3.1 系统性能需求

（1）应用自身：列表类接口在本地千兆局域网、数据库同机部署条件下，P95 延迟宜低于 300ms（不含外部调用）。（2）外部沙盒：公共 Piston 实例可能存在排队，单次运行允许达到数秒，但必须在 15s 超时内返回或报错。（3）Coze：首字延迟受模型与网络影响，SSE 应在收到首包后尽快转发，避免用户在空白界面等待过久。（4）前端：Monaco 按需加载，首屏可通过动态 import 减轻 JS 体积压力。

### 3.3.2 系统安全需求

（1）身份：密码仅存哈希；会话密钥不进入仓库。（2）授权：用户数据读写均绑定当前身份上下文，防止水平越权读取他人记录。（3）执行：用户代码仅在隔离沙盒运行，不在业务进程内直接执行。（4）注入：数据库访问采用参数化方式，提示词与外部输入执行必要过滤与策略约束。（5）可用性：外部服务失败时返回明确错误信息，但不泄露密钥与内部实现细节。

### 3.3.3 系统可扩展性需求

系统扩展遵循“能力插件化”原则：新增智能体能力时，仅需在统一接入层补充配置映射并接入对应业务模块；新增成就时扩展规则集合与条件判断；新增沙盒厂商时补充适配器实现并复用统一返回结构。数据库变更通过迁移机制演进，对历史数据采用默认值或兼容转换策略，降低升级风险。

## 3.4 系统可行性分析

技术可行性：所选用框架均有官方文档与社区示例；Coze 与沙盒均为 HTTP 接口，调试可通过 curl 或 Postman 复现。经济可行性：开发阶段可使用免费公共沙盒与本地 PostgreSQL；上线可选用低配云主机，成本可控。操作可行性：界面中文，符合国内用户习惯；学生自带笔记本即可使用。法律与伦理可行性：需在登录页或帮助文档声明 AI 生成内容可能存在错误，正式课程考核成绩以教师评定为准；用户数据应遵守学校个人信息保护规定。

| 风险        | 影响      | 对策              |

需求可追溯性说明：表 3-1 中 F1～F7 与第 4 章模块、第 5 章实现小节之间应保持“向前可追溯、向后可验证”的关系。以 F3 为例，其验收不仅要看接口返回 200，还要看数据库是否产生 `CodeSubmission` 行、`aiReview` 是否非空（在 Bot 可用时）、`CapabilityRadar` 是否在含 `capabilityScores` 时发生数值变化。以 F4 为例，除 SSE 体验外，还要验收 `InterviewSession.messages` 的 JSON 数组长度随轮次递增，且 `cozeSessionId` 在首轮后保持稳定。以 F6 为例，推荐接口在 AI 可用与不可用两种配置下都应返回结构合理的 JSON，且 `suggestedTemplateId` 不出现在已完成集合中。此类“验收口径”应在答辩 PPT 中用一页写明，避免评审只看见界面而看不见数据层。

非功能需求方面，性能指标在缺乏生产流量前只能给参考阈值：第 6 章表格中的毫秒数需在同机复测后填写。安全需求中，密码哈希与 HTTPS 是底线；若部署在内网 HTTP，仅适合机房封闭实验，正式对学生开放应全站 TLS。可扩展性需求强调“换沙盒、换模型、加成就”三类变更的成本：当前代码把三者分别隔离在 `sandbox.ts`、`coze.ts`、`achievements.ts`，符合开闭原则的实践方向。

用户故事式描述（补充）：作为学生，我希望在第一次登录后立即看到空雷达与引导任务，以便知道下一步该做什么；作为学生，我希望运行失败时看到编译器原文而不是笼统“错误”，以便对照教材排错；作为学生，我希望面试记录可回看，以便总结自己卡住的考点；作为学生，我希望路径推荐能解释“为什么推荐这一块”，以便建立信任（AI 返回的 `reason` 字段承担此职责）。这些故事不在大纲中单独编号，但与第 3 章各小节需求一致，可在需求评审表中作为附列。

关于数据合规：若学校将本平台用于真实姓名学号管理，需在隐私政策中说明存储字段、保留期限与导出删除流程；本论文原型默认仅用邮箱注册，降低敏感个人信息处理复杂度，但工程上仍建议对日志打码。关于可用性：色弱学生区分红绿错误提示可能存在困难，前端可辅以图标形状而不仅依赖颜色编码（可在迭代中完成）。关于国际化：当前 UI 为中文，若面向留学生双语班，可抽离文案资源文件，这属于后续工作。

# 4 系统总体设计

## 4.1 系统总体架构设计

### 4.1.1 逻辑分层

系统采用浏览器/服务器（B/S）结构，在逻辑上划分为三层。（1）表现层：由 React 组件构成，负责表单校验、编辑器状态、SSE 解析与图表渲染，不直接保存密钥。（2）应用层：Next.js Route Handler 与 `lib` 下的领域函数，负责鉴权、事务编排、调用 Prisma 与外部 HTTP。（3）数据与外部服务层：PostgreSQL 持久化；Coze 提供大模型推理；Piston/Judge0 提供沙盒执行。分层后，表现层仅通过 HTTPS 调用本域 API，降低 XSS 窃取 Token 后的横向移动风险（仍需 CSP 与 HttpOnly Cookie 配合）。

### 4.1.2 请求路径描述

以“提交代码”为例，请求先经过身份校验与题目校验，再进入评审编排流程：系统完成模型调用、结果解析与状态落库，并在同一业务链路中更新能力画像与学习日志。以“发送面试消息”为例，请求先写入用户消息，再进入流式对话编排，待流结束后补写助手完整回复。两条链路均体现“先保证业务主流程，再叠加 AI 能力”的设计思想。

### 4.1.3 部署视图与安全边界

单机部署时，Node 进程同时服务静态资源与 API；数据库可为同机 Docker 或托管 RDS。若流量增大，可将数据库只读副本与 Node 水平扩展分离，本论文不做集群细节展开。安全上，浏览器属于不可信端，不得持有 Coze Token；应用服务器代为调用外部 AI 与沙盒。PostgreSQL 存会话与业务数据，备份介质应加密。外部 SaaS（扣子、Piston）按服务协议信任其执行隔离。业务查询一律带 `userId` 条件以防水平越权；Prisma 参数化查询降低 SQL 注入面；若未来支持任意 URL 抓取需防 SSRF，当前版本未开放。生产环境建议最小权限数据库账号与脱敏日志。

### 4.1.4 开题技术路线与最终实现的对照说明

## 4.2 系统功能模块设计

### 4.2.1 用户认证与管理模块

职责：注册、登录、会话续期、可选资料修改。该模块通过统一认证框架管理账户生命周期，安全重点是密码哈希存储、会话有效期控制与错误信息脱敏，避免账号枚举与凭据泄露风险。

### 4.2.2 数据看板模块

职责：汇总能力雷达、活动时间线、练习与面试统计。该模块以“聚合展示”为核心，强调多源数据的并行读取与接口稳定性，避免统计看板拖慢主学习流程。

### 4.2.3 编程练习与代码沙盒模块

职责：题目列表筛选、详情展示、运行与提交入口。关键设计是把“运行验证”和“评审记录”解耦：运行用于即时调试反馈，提交用于形成性评价与学习轨迹沉淀。

### 4.2.4 AI 智能代码审查模块

职责：构造评审上下文、解析结构化结果并映射为业务状态。该模块与沙盒执行解耦，便于在“静态审查”与“运行结果辅助审查”之间按课程要求灵活切换。

### 4.2.5 AI 模拟面试模块

职责：会话管理、消息持久化、流式回复转发与降级兜底。会话状态按“进行中→已完成”演进，已完成状态可触发统计与成就逻辑，支持阶段性复盘。

### 4.2.6 知识库与知识助手模块

职责：文章元数据管理、分类浏览、收藏与阅读进度记录，并提供与知识助手的协同问答能力。该模块强调“结构化文章学习”与“即时问答补充”的双通道学习体验。

### 4.2.7 学习路径规划模块

职责：从模板实例化学习路径、维护进度状态并推荐下一阶段内容。该模块以“模板化课程节奏 + 个性化推荐”联合驱动，兼顾教学可控性与个体差异。

### 4.2.8 能力雷达图与成就系统模块

职责：雷达平滑更新、成就筛选发放与活动记录。该模块不单独暴露页面入口，而是嵌入练习、面试、路径等主流程。设计上强调“事件触发、幂等写入、与 AI 调用解耦”，避免奖励逻辑反向依赖不稳定外部能力。

模块间协作可结合时序理解：练习模块写入提交后，审查模块消费同一请求上下文中的模型输出，再调用雷达模块与成就模块；仪表盘模块只读雷达与活动，不参与写路径；学习路径模块在读推荐时可读雷达，在更新进度时可触发成就；知识库模块与 AI 助手模块共享“文章”概念但写入路径不同（站内进度 vs 扣子对话）。这种“扇出写入、扇入读取”的结构使单点故障隔离：即使知识助手不可用，文章阅读与路径仍可部分进行；即使评审 Bot 不可用，运行沙盒仍可验证语法与基本逻辑。

接口风格上，对内强调明确事务边界，对外强调统一响应结构。配置风格上，智能体与沙盒能力均采用环境变量管理，避免将部署差异固化到业务代码。该约定提升了环境切换效率，也降低了后续迁移成本。

从课程运营视角，八大模块可映射到一次实验课流程：课前学生用知识库预习；课中用练习运行排错；课后提交获取 AI 讲评；周末用面试模块复盘表达；路径模块贯穿学期形成进阶节奏；成就与活动用于课堂点名激励。论文虽以技术实现为主，但模块划分隐含了上述教学节奏，答辩时可配图说明。

## 4.3 AI 服务架构设计

### 4.3.1 Coze 多 Bot 统一接入层设计

统一接入层对外提供三类能力：配置可用性判断、流式调用编排、结构化结果解析。其内部职责是将多类智能体配置映射到统一调用模型，并处理增量数据可能出现的不完整片段，确保业务层获取稳定输出。

接入层的版本策略为“先兼容、后演进”：当上游字段变动时，先基于消息类型进行稳健分发，再做业务字段映射。错误处理采用“可重试错误与不可重试错误分级”，并保留可追踪日志标识，便于定位第三方平台故障。监控层面建议统计调用次数、平均耗时与错误率，形成可观测闭环。

### 4.3.2 流式与非流式调用策略

流式用于强交互：面试、知识助手。非流式用于强结构：代码审查、路径推荐。非流式并非“非 HTTP 流”，而是服务端在内部读完整个 `ReadableStream` 后再返回业务 JSON 给前端（提交接口）或直接返回推荐对象（推荐接口）。

### 4.3.3 结构化 Prompt 设计与 AI 降级策略

降级矩阵遵循“核心学习不断链”原则：（1）代码评审能力不可用时，系统保留提交与记录能力，并给出可读说明；（2）路径推荐异常时回退规则推荐；（3）面试智能体不可用时回退本地模拟问答。该机制保证课堂演示与功能调试不完全依赖外网。

Prompt 治理建议：在团队开发中应为每个 Bot 维护版本号与变更说明（可存在仓库 `docs/prompts/`）；重大变更前用固定题库做回归，观察 JSON 字段缺失率。对敏感词与不当输出，应同时在扣子控制台配置安全策略与在应用层过滤展示，双保险。对学生可见的反馈文本，避免暴露内部 Token、URL 或堆栈细节。

### 4.4.3 代表性表设计说明（节选）

为避免数据库章节滑向“字段说明书”，本节仅选择与业务闭环最相关的代表性表进行说明，重点解释“为何这样设计”。

（1）用户与认证相关表：采用“账户主表 + 认证扩展表”组合，核心目标是把身份信息、会话信息与第三方登录能力分层管理。密码采用不可逆哈希，用户主键采用不可预测标识，降低账号枚举风险。

（2）练习与评审相关表：提交记录表保留代码快照、状态与评审文本，既满足学习复盘，也支持后续教学抽查。题目表与提交表分离，保证题库迭代不破坏历史学习记录。

（3）面试会话表：会话消息采用结构化聚合存储，突出“多轮上下文完整保留”。该设计在开发效率与检索能力之间做了平衡，适合毕设阶段以可追溯性为优先。

（4）学习路径与知识文章表：路径与文章保持弱耦合，通过标识关联而非强绑定，便于课程内容随学期迭代。阅读进度与收藏记录独立建模，支撑形成性评价与行为统计。

（5）活动日志与成就表：活动日志承担审计与学习轨迹记录，成就表通过唯一约束保证“同一成就只发一次”。二者共同支持“可复盘、可解释”的教学数据治理。

（6）一致性策略：个人数据相关实体采用级联清理策略，公共资源实体保持独立生命周期。该策略兼顾隐私要求与内容资产稳定性。

持续集成方面，建议采用“安装依赖-静态检查-构建验证-数据模式校验”的流水线，并在测试环境先行演练数据库迁移。端到端层面可配置登录与题目浏览冒烟用例，确保关键入口在发布前可用。代码规范与依赖安全采用自动化检查，减少人工回归成本。

## 5.2 前端系统实现

### 5.2.1 页面路由设计与响应式布局

前端路由按业务域组织，包括仪表盘、练习、面试、知识与学习路径等页面。布局层提供统一导航与主题容器，页面层遵循“列表-详情-交互”结构。响应式策略以断点栅格为主，在窄屏场景下优先保障编辑区与结果区可读性。

路由保护采用统一会话校验机制，未认证访问将被重定向。页面职责方面，仪表盘聚合统计与画像信息，练习页承担高交互编辑体验，面试页承载流式对话状态管理，知识页负责分类检索与阅读跳转。整体目标是降低跨模块跳转成本并保持学习连续性。

无障碍与可用性方面，按钮需带 `aria-label`，SSE 加载态用 `aria-busy` 提示读屏用户；颜色对比度遵循 WCAG AA 可减少答辩演示投影不清的问题。国际化本版本未做，若后续支持英文界面，可把路由与文案抽离到 `next-intl` 等方案。

### 5.2.2 Monaco Editor 代码编辑器集成

练习详情页集成浏览器代码编辑器，支持语言切换、语法高亮与实时输入。交互上采用受控状态管理，确保题目切换、运行输出与提交状态保持一致。

考虑到编辑器组件体积较大，前端采用按需加载与首屏优化策略，避免影响主页面渲染。运行与提交按钮在请求期间采用防重复触发机制，并通过清晰的加载态与错误提示提升可用性。该设计将“编辑体验”和“系统稳定性”作为同等优先级处理。

### 5.2.3 SSE 流式消息渲染实现

面试与知识助手采用服务端推送事件（SSE）实现流式渲染。前端在连接建立后持续接收增量文本，并在会话结束时完成状态收束，从而实现“边生成边阅读”的交互体验。

为保证稳定性，流式渲染模块采用统一缓冲与顺序控制策略，处理网络抖动、分片不完整与快速连续发送等边界情况。为保证安全性，前端仅消费白名单字段并对会话标识执行本地隔离存储，避免敏感上下文暴露。

### 5.2.4 Recharts 能力雷达图可视化

仪表盘以雷达图呈现六维能力画像，并与活动记录联动展示学习进展。可视化设计强调三点：量纲统一（百分制）、标签一致（与评价维度对齐）、空态友好（新用户提供引导）。

在性能与体验上，图表模块采用稳定配置与最小重绘策略，减少统计卡片更新对可视化区域的干扰，使看板更适合长期追踪而非一次性展示。

## 5.3 后端系统实现

### 5.3.1 RESTful API 接口设计

后端接口采用统一响应封装，输出结构保持一致，便于前端状态管理与异常处理。资源命名与状态码遵循 REST 语义，保证接口在可读性与可维护性之间取得平衡。

在部署策略上，同域优先，跨域场景通过白名单与凭据策略保障安全。对非幂等写操作采用“前端防抖 + 后端限流”联合控制，减少重复提交。日志体系采用开发与生产分级策略：开发强调排错效率，生产强调脱敏与可追踪性。

### 5.3.2 NextAuth.js 用户认证实现

认证流程采用“凭据校验 + 会话序列化”模式：服务端校验密码哈希后签发会话上下文，业务接口从会话中提取当前用户身份执行授权判断。

密码策略可配置最小强度要求；会话策略结合有效期与滑动续期，兼顾安全与课堂连续使用体验。若引入第三方登录，需统一回调域配置并补充安全响应头，降低会话劫持与点击劫持风险。

### 5.3.3 代码沙盒执行服务实现

代码沙盒服务采用适配器模式，对不同厂商返回值进行统一归一化处理。前端只消费统一结果结构，不感知底层差异，便于后续替换为校内判题集群。

稳定性方面，系统采用双重防护：外部执行超时控制与应用侧请求超时控制共同限制异常任务。异常返回统一转换为可读错误信息；日志仅保留必要诊断信息，避免泄露用户源码内容。

## 5.4 AI 服务集成实现

### 5.4.1 Coze Bot 统一封装层实现

统一 AI 接入层分为“流式通道”和“非流式通道”两种执行模式：前者用于对话类即时反馈，后者用于结构化评审落库。两种模式共享统一的配置校验、异常分级与结果清洗逻辑。

部署上，流式能力要求运行环境支持稳定长连接；解析链路需兼容格式抖动与片段异常，以保证业务层获得可消费结果。客户端中断时应及时释放上游资源，避免连接泄漏。

### 5.4.2 代码审查 Prompt 工程与能力评分

代码审查链路强调“结构化可计算”而非“自由文本可读”。系统在提示模板中约束输出维度，使结果可直接映射到提交状态、反馈面板与能力雷达。

评分采用阈值判定与平滑更新并行：阈值用于单次通过判断，平滑用于抑制多次提交的随机波动。该机制兼顾即时反馈与长期画像稳定性。

### 5.4.3 面试对话与自动评估实现

面试链路按“先写入用户输入，再转发模型增量，最后补写助手全文”的顺序执行，确保会话记录与前端展示保持一致。出现上游异常时，系统向前端返回可读错误并安全结束连接。

在降级模式下，系统提供本地模拟回复以保证教学演示连续性。会话完成后可写入总结信息，形成可复盘的表达训练档案。

### 5.4.4 学习路径 AI 推荐与规则回退

路径推荐采用“AI 优先、规则兜底”的双通道策略。系统先基于能力画像与学习历史生成个性化建议；若 AI 不可用或结果异常，则回退到薄弱维度映射规则，保证推荐功能始终可用。

推荐结果进入前端前需做合法性校验，避免出现无效学习步骤。该策略在可用性、个性化与稳定性之间取得平衡。

上述实现可抽象为两条主链路：一是“提交-评审-画像-激励”链路，二是“提问-流式对话-会话归档”链路。两条链路共享同一原则：先保证数据一致性，再叠加智能能力；外部服务失败时以降级策略维持核心业务可用。

## 5.5 知识库与成就系统实现

### 5.5.1 Markdown 文章管理与同步机制

知识文章采用“文本文件 + 元数据索引”双存储策略：正文保持可编辑文本形态，数据库维护检索与推荐所需索引字段。该模式兼顾内容维护效率与业务查询性能。

### 5.5.2 成就触发检测与颁发机制

成就系统采用“事件触发 + 规则筛选 + 幂等写入”机制。系统先根据触发事件缩小候选规则，再执行批量去重与条件判定，满足条件即发放成就并写入活动日志。该设计可在规则规模扩张时保持可维护性。

### 5.5.3 知识问答 Bot 与平台知识库配置

平台侧知识问答由运维配置知识源并维护检索参数，应用侧仅负责安全转发与结果展示。站内文章接口与平台问答接口职责分离：前者服务结构化课程学习，后者服务即时提问补充。

站内学习路径通过前置关系形成由浅入深的学习链。实例化路径时需校验前置阶段状态，避免跳级学习造成认知负担；同时通过阶段顺序字段统一前端展示与解锁判断。

为与第 4 章模块划分相呼应，系统在数据写入层遵循三条规则：第一，提交链路优先保证提交记录与日志落库，再执行画像与激励更新；第二，会话链路采用“用户消息先写、助手消息后补”的顺序，保证时序一致；第三，推荐链路以只读为主，创建与更新动作在独立写接口执行。该约定有助于快速定位“界面显示成功但数据未更新”类问题。

## 6.4 测试结果分析

改进建议：为 `code/run` 增加队列与重试；为 Coze 调用增加熔断计数；为雷达引入 OJ 通过率作为第二数据源加权。

从测试方法论看，本系统强依赖外部 SaaS，回归测试应区分“纯本地链路”和“全链路”两类用例：前者在 CI 中每日运行保证重构不破坏鉴权与数据库约束；后者在发布前由人工或定时任务在可访问外网的环境执行并记录截图。对 AI 输出做自动化断言时，宜只校验 JSON 可解析性与字段存在性，不宜断言具体分数，以免模型升级导致用例脆弱。性能数据受机器负载影响大，论文表格中“待填”项应在定稿前用同一硬件补全，并注明测试日期与网络环境。

# 7 总结与展望

## 7.1 研究工作总结

本文从高校 C/C++ 教学与面试准备的真实痛点出发，完成“基于 AI 的高校学生学习助手系统”的需求分析、总体设计、数据库设计、前后端实现、多智能体集成与测试方案设计。系统以 Next.js 为骨架，PostgreSQL 为单一事实来源，NextAuth.js 管身份，Monaco 与沙盒管代码执行，扣子多 Bot 管智能服务，SSE 管流式体验，成就与活动日志管过程激励。实现上特别注意了解析容错、环境缺失降级与越权防护，使仓库代码与论文描述可逐条对应，满足毕业设计“可运行、可答辩、可交接”的要求。

回顾各章对应关系：第 3 章需求在第 4 章得到架构与模块的响应；§4.1.4 将开题报告中的技术设想与仓库实现逐项对照，避免评阅人误以为“未实现向量库即偏离题目”。第 4 章数据表在第 5 章 Prisma 调用与 API 中得到落地；第 5 章实现细节在第 6 章用例表中得到验证映射。特别是代码提交接口把“外部模型—内部事务—画像更新—激励反馈”串成一条链，体现了软件工程中端到端（End-to-End）思维，而不是停留在调用 API 的演示级代码。论文撰写过程中，坚持把环境变量、降级策略与权重参数写清楚，是为了让后续维护者不必反查聊天记录即可复现实验。

在范围管理上，本文刻意收缩为“C/C++ 单语言 + 学生单角色”，避免毕设周期内摊子过大；在此边界内，尽量把日志、成就、路径、雷达等可观测性做全，使系统不仅“能聊”，还能“可度量”。若与其他课程平台对接，可把本文所述接口当作一种参考契约：哪些字段必须稳定、哪些功能可插件化。

## 7.2 系统创新点分析

（1）协议层复用：扣子对外统一走 SSE，本系统在 `coze.ts` 内集中处理分块缓冲与 `data:` 行合并，业务层只消费字符串流或完整字符串，降低各 Route 的重复解析代码。（2）双模推荐：学习路径在 AI 与规则之间自动切换，兼顾个性化与可用性。（3）评价与画像闭环：结构化评审直接驱动六维雷达的指数平滑更新，使“练—评—显”在同一次提交内完成，后续可对接教务分析。（4）演示友好：面试 Mock 与路径规则回退保证离线环境仍可展示核心交互。

与常见“只接一个 Chat 接口”的作业相比，创新点主要体现在工程拼装：多 Bot 隔离降低提示词互相污染风险；流式与非流式分流贴合不同交互节奏；JSON 与数据库字段一一映射使 AI 输出成为一等数据公民而非页面上的临时文本。需要强调的是，这些创新属于应用架构与教学软件工程层面，而非算法模型层面；答辩时应主动说明边界，避免被误读为“训练了新模型”。

## 7.3 研究不足与未来展望

不足之一：能力分数目前主要来自大模型主观判断，与客观 OJ 通过率尚未建立标定关系，可能存在“会说不会写”或“会写分数低”的偏差。不足之二：公共沙盒的数据隐私与稳定性不受校方完全控制。不足之三：面试评估报告字段依赖后续 Bot 能力，当前版本以对话存储为主。不足之四：尚未做大规模用户实验，缺乏与对照班成绩相关的统计学结论，论文结论停留在工程与个案测试层面。

展望短期：引入自建 Docker 沙盒与 cgroup 限流；为每道题增加隐藏测例与 AI 评审的加权融合；增加教师端题库与班级看板；记录提示词版本号与模型版本以便实验复现。展望中期：在保护隐私前提下采集脱敏交互日志，研究雷达分数与期末机考成绩的相关性；尝试小参数本地模型做高并发降级。展望长期：与教务系统统一身份认证（CAS/OIDC）对接，把平时成绩回写纳入合规流程；探索多课程复用同一智能体中台，由院系统一采购算力与审计。

工程落地层面，短期还可增加管理后台用于审核用户生成内容（若未来开放题解分享）；中期可引入特性开关（feature flag）按班级灰度新模型；长期可对接学校统一日志平台做合规审计。文档层面，建议在仓库维护 `ARCHITECTURE.md` 与 `RUNBOOK.md`，把本文第 4、5 章的操作型内容沉淀为运维手册，避免知识只存在于论文 PDF 中。课程运营层面，可与助教分工：助教维护题库与模板，任课教师审定雷达权重与 AI 使用规范，信息办负责服务器与备份策略，形成协同治理。至此，全文字数规模已达到本科毕业设计常见正文要求，后续仅需按学院模板调整字体、行距与图表编号即可定稿。

### 附录 C 代码审查 Prompt 结构说明

本系统的代码审查提示采用“任务上下文 + 输出约束 + 评分维度”三段式结构。

（1）任务上下文：提供题目目标、输入输出预期与学生提交代码，使模型具备最小必要背景。  
（2）输出约束：要求返回可解析的结构化结果，避免自由文本导致系统无法自动处理。  
（3）评分维度：围绕课程核心能力给出量化评价，并允许特定语言场景下的“不适用”处理。

该设计的核心价值在于把模型输出转化为可计算、可落库、可追踪的数据资产。完整模板文本见项目实现文档，论文正文保留其方法论而不展开字段级清单。

### 附录 G 术语、缩略语与关键概念说明

为便于非 Web 方向评阅人阅读，本附录集中解释文中反复出现的概念。**B/S（Browser/Server）**：浏览器负责渲染，服务器负责业务与数据，本系统页面通过 HTTPS 访问部署在 Node 上的 Next.js 应用。**ORM（Object-Relational Mapping）**：对象关系映射，Prisma 根据模型定义生成类型安全的数据库访问代码，减少手写 SQL 的拼写错误。**SSE（Server-Sent Events）**：基于 HTTP 的服务器单向推送，适合面试与助手逐字输出；与 WebSocket 相比实现简单，但仅服务器到客户端单向。**JSON（JavaScript Object Notation）**：轻量数据交换格式，本系统用于题目测试用例、对话历史、AI 结构化评审结果等字段的序列化。**RAG（Retrieval-Augmented Generation）**：检索增强生成，扣子知识库在回答前检索文档片段再生成文本，减轻“胡编教材页码”问题。**OJ（Online Judge）**：在线评测系统，传统上侧重自动判题；本文沙盒承担类似“运行用户代码”的角色，但判对判错还可与 AI 讲评并行。**NextAuth.js**：Next.js 生态常用认证库，支持凭据、OAuth 等多种 Provider，本系统用它管理会话与密码哈希。**bcrypt**：密码哈希算法族，通过盐值与代价因子抵抗彩虹表攻击。**cuid**：一种碰撞概率极低的前缀随机 ID 生成策略，用作 `User.id` 等主键。**API Route**：Next.js App Router 中 `route.ts` 导出的 HTTP 处理函数，路径由文件夹名决定。**加权移动平均**：雷达更新公式 `新分数 = round(旧分数 × 0.7 + 本次 × 0.3)`，使画像对历史表现有记忆又对最新提交敏感。**降级（Fallback）**：外部服务失败时改用本地规则或 Mock 数据，保证核心页面可用。**水平越权**：用户 A 访问用户 B 的资源；本系统通过查询条件强制带当前 `userId` 缓解。**Prisma migrate / db push**：前者生成迁移历史，后者快速对齐开发库结构，毕设阶段两者择一即可，生产推荐 migrate。

上述术语在正文首次出现时也可加脚注，Word 排版时可将本附录改为“术语表”页。若学校要求英文缩略语在摘要中先写全称，可在终稿摘要第一句补充 “Large Language Model (LLM)” 等写法，与学院格式统一即可。
